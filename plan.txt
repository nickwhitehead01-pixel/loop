OVERVIEW
--------
A 100% offline, privacy-first Edge AI ecosystem for special education classrooms.
Every student gets a personalised AI tutor. Every teacher gets a smart teaching
companion. Not a single byte of student data ever leaves the school network.

Four-Node Architecture:

  1. WINDOWS HUB (Server Node)
     - IT admin installs a 1-Click launcher on a school Windows Desktop
     - Silently starts: Ollama (local GPU/CPU) + Docker Compose stack
     - Stack: FastAPI + LangGraph backend + PostgreSQL/pgvector
     - Runs faster-whisper for live classroom speech-to-text
     - Broadcasts mDNS (Bonjour/ZeroConf) for auto-discovery on local network

  2. TEACHER COMPANION (Command Node)
     - Native iOS/iPadOS app on teacher's school iPad
     - Pre-lesson: upload lesson plans, PDFs, worksheets → synced to Hub
     - Live lesson: iPad acts as high-fidelity wireless mic → streams audio
       to Hub → generates live captions + AI context for all students

  3. PUPIL CLIENT (Edge Nodes)
     - Cross-platform pupil app (iPad + Android phones/tablets incl. Pixel)
     - Auto-discovers Windows Hub via mDNS/NSD with manual URL fallback
     - Each student has their own personalised LangGraph ReAct agent:
         Short-term memory:  sliding window of recent messages
         Long-term memory:   atomic learning facts (pgvector embeddings)
         RAG context:        live transcript + uploaded lesson materials
     - Accessibility: on-device TTS + simplified UI for neurodivergent UX

  4. INFRASTRUCTURE (Air-Gap)
     - All traffic stays on school Wi-Fi (WLAN) — zero internet egress
     - Inherently FERPA / GDPR / IEP / COPPA compliant by design

Stack:
  - Backend:    FastAPI + LangGraph (Python)
  - Database:   PostgreSQL + pgvector
  - LLM:        Ollama (local) — gemma4:e2b
  - Embeddings: nomic-embed-text
  - STT:        faster-whisper (CPU, int8)
  - Hub OS:     Windows (Docker Desktop)
  - Clients:    SwiftUI teacher app (iPad) + Flutter pupil app (iPad + Android)
  - Discovery:  mDNS / Bonjour / ZeroConf + Android NSD fallback

Model Strategy:
  - Primary:  gemma4:e2b  (pupil chat + teacher RAG, good balance of capability & memory)
  - Fallback: gemma4:e2b  (same as primary for consistency)
  - Embeddings: nomic-embed-text (768-dim, fully local)
  - STT:        faster-whisper tiny/small (CPU, int8, real-time)

PHASE 7b — SEMANTIC ANSWER CACHE                               [DONE]
----------------------------------------------------------------------

In a classroom of 30 pupils, many students ask semantically identical
questions about the same lesson. Running Gemma 4 per-student wastes
cycles on constrained school hardware. A per-lesson semantic cache
solves this.

Schema (new table: semantic_cache)
  id            UUID PK
  lesson_id     FK → lessons.id
  question      TEXT
  question_emb  Vector(768)        -- nomic-embed-text embedding
  answer        TEXT
  hit_count     INT DEFAULT 0      -- for analytics / popularity
  created_at    TIMESTAMP

Lookup flow (services/semantic_cache.py)
  lookup(question: str, lesson_id: UUID, threshold: float = 0.92)
    → embed question via nomic-embed-text
    → cosine similarity search against semantic_cache WHERE lesson_id = ?
    → if best_score >= threshold: increment hit_count, return answer
    → else: return None

  store(question: str, answer: str, lesson_id: UUID)
    → embed question
    → insert row into semantic_cache

Wiring into pupil_graph.py
  - Before invoking the LangGraph agent: call lookup()
  - On cache hit: stream cached answer directly (no Ollama call)
  - On cache miss: run agent normally, then call store() with result

Cache lifecycle
  - Scoped per lesson_id — answers stay relevant to current material
  - Cache cleared / archived when lesson session ends
  - Threshold configurable via Settings (semantic_cache_threshold: float)

Analytics benefit
  - hit_count exposes which questions the class is asking most
  - Teacher dashboard can surface "top questions this lesson"
